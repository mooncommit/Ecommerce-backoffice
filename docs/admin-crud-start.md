# 관리자 CRUD 구현을 시작하기 전에

저번 글에서는 팀원들과 공통 구조와 협업 방식을 맞추는 작업을 정리했었다. 이제는 각자 맡은 도메인을 구현해야 하는 단계가 되었다.  
이번 글에서는 내가 맡은 관리자 CRUD 기능을 시작하면서, 무엇부터 구현해야 할지 어떻게 생각하고 정리했는지 기록해보려고 한다.

...

## "구현을 시작하기 전에"

기능 목록이 정리되어 있어도 막상 구현을 시작하려고 하니 무엇부터 손대야 할지 막막했다.  
그래서 바로 코드를 작성하기보다, 관리자 기능을 어떤 기준으로 보고 어떤 순서로 시작해야 하는지 먼저 정리해보려고 한다.

---

## ✔️ 과제 문제를 보면서 먼저 기능을 나누어 보기

관리자 파트 구현을 시작하기 전에 가장 먼저 한 것은, 과제 문제를 기능 단위로 다시 나누어 보는 일이었다.  
처음에는 요구사항이 길게 적혀 있어서 해야 할 일이 많아 보였지만, 큰 항목과 세부 항목을 기준으로 나누어 보니 생각보다 흐름이 더 명확하게 보였다.

과제 문제는 크게 아래 3가지로 나뉘어 있었다.

1. 관리자 회원가입
2. 관리자 인증
3. 관리자 정보 관리

이 큰 항목들을 기준으로 다시 내가 구현해야 할 기능을 정리해보았다.

### 1. 관리자 회원가입
- 관리자 회원가입 신청

### 2. 관리자 인증
- 관리자 로그인
- 관리자 로그아웃

### 3. 관리자 정보 관리
- 관리자 승인 / 거부
- 관리자 목록 조회
- 관리자 상세 조회
- 관리자 정보 수정
- 관리자 역할 수정
- 관리자 상태 수정
- 관리자 삭제
- 내 프로필 조회
- 내 프로필 수정
- 비밀번호 변경

이렇게 기능을 다시 나누어 보니, 처음에는 길게만 보이던 요구사항이 조금 더 구체적으로 보이기 시작했다.  
특히 문제를 읽을 때는 문장을 그대로 따라가기보다, 큰 항목과 세부 항목을 기준으로 내가 구현해야 할 기능을 다시 정리해보는 과정이 필요하다는 걸 느꼈다.

---

## ✔️ 그러면 무엇부터 구현해야 할까

기능을 정리하고 나니 그다음에는 자연스럽게 구현 순서를 생각하게 되었다.  
이때 알게 된 점은 기능 정리와 구현 순서를 정하는 기준은 조금 다르다는 것이었다.

기능을 정리할 때는 문제의 큰 항목과 세부 항목을 기준으로 나누면 되지만,  
구현 순서를 정할 때는 각 기능이 어떤 데이터를 필요로 하는지, 어떤 기능이 먼저 있어야 하는지를 함께 봐야 했다.

그래서 나는 아래 기준으로 다시 생각해보았다.

- 이 기능이 실행되려면 먼저 어떤 데이터가 있어야 하는가
- 이 기능은 다른 기능의 결과를 필요로 하는가
- 로그인한 사용자가 꼭 필요한 기능인가

이 기준으로 다시 보니 관리자 기능은 아래 흐름으로 이어져 있었다.

- 회원가입을 해야 관리자 데이터가 생성된다.
- 회원가입한 관리자는 바로 사용 가능한 것이 아니라 `승인대기` 상태가 된다.
- `승인대기` 상태의 관리자가 있어야 승인 / 거부가 가능하다.
- 승인되어 `활성` 상태가 되어야 로그인할 수 있다.
- 로그인한 사용자가 있어야 내 프로필 조회나 비밀번호 변경이 가능하다.

이 흐름으로 보니 무엇부터 만들어야 하는지도 조금 더 분명하게 보였다.

1. 관리자 회원가입 신청
2. 관리자 승인 / 거부
3. 관리자 목록 조회 / 상세 조회
4. 관리자 정보 수정 / 역할 수정 / 상태 수정 / 삭제
5. 내 프로필 조회 / 수정
6. 비밀번호 변경

즉, 먼저 데이터를 만들고, 그다음 그 데이터를 조회하고, 이후 상태를 변경하는 기능을 구현한 뒤, 마지막에 로그인한 사용자 기준 기능을 붙이는 흐름으로 생각하게 되었다.

---

## ✔️ 이번에 느낀 점

이번 과제를 통해 느낀 점은, 기능 구현 순서를 정하는 것도 하나의 설계라는 점이었다.  
예전에는 요구사항을 보면 눈에 먼저 들어오는 기능부터 구현하려고 했는데, 이번에는 각 기능이 어떤 조건 위에서 동작하는지 먼저 생각해보게 되었다.

특히 관리자 도메인은 회원가입을 해도 바로 로그인할 수 있는 구조가 아니라, 승인 절차와 상태 변화가 함께 있는 도메인이었기 때문에 단순 CRUD처럼만 보면 안 된다는 것도 느꼈다.

이제는 과제 문제를 보면 단순히 어떤 API를 만들어야 하는지만 보는 것이 아니라,  
이 기능이 동작하려면 먼저 무엇이 있어야 하는지를 함께 생각해야 한다는 걸 조금은 알게 된 것 같다.
